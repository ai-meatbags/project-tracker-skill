# Архитектурные паттерны

Дата: 2026-03-07

## Назначение
Этот документ хранит активные архитектурные правила baseline-уровня для проектов, инициализируемых через project-tracker.

Baseline ориентирован на веб-приложения с базами данных, авторизацией и требованием к предсказуемой масштабируемости.

Правила здесь делятся на:
- общие архитектурные правила;
- условные стековые правила, явно помеченные областью применимости.

`tasks/_policies/arch-patterns.md` обновляется только upstream-деплоем и не редактируется локально в рамках выполнения задач.

Процесс реализации, Definition of Done и CI-flow описываются в `tasks/_policies/dev-plan.md`.
Технические ограничения конкретной фичи фиксируются в feature-spec в отдельной секции `Технические ограничения реализации`, но не являются источником истины для архитектурных отклонений.
Репозиторные технические принципы и допустимые отклонения от baseline фиксируются отдельно в `tasks/_policies/project-patterns.md`.

## Приоритет и разрешение конфликтов
1. `critical` — архитектурный инвариант. Нарушение требует изменения архитектуры, а не локального компромисса в реализации.
2. `baseline` — дефолт проекта. Отклонение допускается только при явном техническом обосновании и фиксируется в `tasks/_policies/project-patterns.md`.
3. `guideline` — сильная инженерная рекомендация. Она направляет ревью и дизайн решения, но сама по себе не блокирует проектное решение.
4. `Этап` показывает, с какого момента правило должно считаться обязательным для проекта: `mvp`, `prod`, `scale`.
5. Feature-spec не может отменять `critical` архитектурные правила; он может только ссылаться на уже зарегистрированное baseline-отклонение из `tasks/_policies/project-patterns.md`.
6. Условные стековые правила обязательны только в пределах явно указанной применимости.
7. Нумерация AP-### стабильна и не переиспользуется, даже если правило перенесено, архивировано или снято с активности.

## Семантика этапов

- `mvp` — первый живой продукт, где уже важны корректность ядра, безопасность границ и предсказуемость архитектуры.
- `prod` — система реально эксплуатируется, и сбои, деградация, утечки или неготовность уже имеют заметную операционную цену.
- `scale` — рост нагрузки, данных, интеграций или команды делает performance, operability и database guardrails регулярным ограничением.

Правила чтения этапов:
- `Этап` не является фильтром просмотра правил: при проектировании feature команда обязана просматривать весь набор AP, а не только правила "своей текущей стадии".
- `Этап` показывает порог, с которого правило становится неотложно обязательным по умолчанию.
- Если новая feature может фактически перевести проект из `mvp` в `prod` или из `prod` в `scale`, это должно быть явно рассмотрено на стадии `Context/Spec/Plan`, а не только после реализации.
- Переход стадии считается монотонным: `mvp -> prod -> scale`.

## Структурные паттерны

### AP-010 — Модульный монолит + Ports & Adapters
**Добавлено:** 2026-02-12 13:30  
**Описание:** Бизнес-логика изолируется в модулях и зависит только от контрактов, а не от деталей интеграций.  
**Почему:**
- Ускоряет локальные изменения и уменьшает связность между частями системы.
- Позволяет менять интеграции без переписывания бизнес-правил.
- Делает тестирование доменной и прикладной логики независимым от I/O.  
  **Требование:**
- Доменная и прикладная логика не импортирует инфраструктурные реализации напрямую.
- Интеграции (БД, HTTP, внешние API, очереди) подключаются через адаптеры.
- Связывание конкретных адаптеров выполняется только в composition root.  
  **Критерии проверки:**
- Use-case можно запустить с подменой адаптера без реальной интеграции.
- Замена реализации порта не требует изменения бизнес-правил.
- Внутри core/application нет прямых импортов конкретных DB/API клиентов.

### AP-011 — Низкая связность и высокая когезия
**Добавлено:** 2026-02-12 13:30  
**Описание:** Каждый модуль отвечает за один согласованный бизнес-контекст; межмодульные зависимости минимальны и проходят через узкие контракты.  
**Почему:**
- Упрощает локальные изменения и тестирование.
- Снижает каскад регрессий и стоимость поддержки.
- Делает ownership модуля понятным на ревью и в эксплуатации.  
  **Требование:**
- Модуль содержит только код, относящийся к одному бизнес-сценарию.
- Публичная поверхность модуля минимальна и стабильна.
- Межмодульные вызовы идут через явные интерфейсы/контракты, а не через внутренние детали другого модуля.  
  **Критерии проверки:**
- Модуль можно кратко описать одной бизнес-ответственностью.
- Для типового изменения не требуется правка большого числа несвязанных модулей.
- Межмодульный вызов не использует внутренние файлы или скрытые helper-детали чужого модуля.

### AP-014 — SOLID и Clean Code на уровне модулей
**Добавлено:** 2026-02-12 13:30  
**Описание:** Код проектируется для читаемости, локальности изменений и расширяемости без переписывания существующих веток.  
**Почему:**
- Система дольше сохраняет предсказуемую структуру по мере роста.
- Новые сценарии проще добавлять через расширение, а не через ломку старого кода.
- Ревью и сопровождение становятся дешевле.  
  **Требование:**
- Single Responsibility: один модуль/класс/функция — одна причина изменения.
- Open/Closed: расширение через новые use-case/адаптеры, а не через разрастание условной логики.
- Dependency Inversion: high-level правила не зависят от low-level деталей.
- Имена, контракты и ошибки должны быть самодокументируемыми.  
  **Критерии проверки:**
- Новое поведение чаще добавляется новой веткой композиции, а не каскадом `if/switch` в старом модуле.
- High-level код не знает конкретных реализаций интеграций.
- Название модуля и его публичный API отражают бизнес-роль, а не случайную технику реализации.

### AP-015 — DRY, KISS, YAGNI
**Добавлено:** 2026-02-12 13:30  
**Описание:** Реализация должна быть минимально достаточной, без дублирования и без преждевременной абстракции.  
**Почему:**
- Снижает сложность и ускоряет delivery.
- Уменьшает стоимость последующих изменений.
- Защищает кодовую базу от "архитектуры на вырост" без реального спроса.  
  **Требование:**
- DRY: не дублировать бизнес-правила; выносить общую логику после появления повторяемого паттерна.
- KISS: предпочитать простые решения со стабильным поведением и малым числом вариантов исполнения.
- YAGNI: не реализовывать расширения и инфраструктуру "на будущее" без текущего требования.  
  **Критерии проверки:**
- Новая абстракция появляется из повторяемой необходимости, а не из предположения о будущем.
- Одно и то же бизнес-правило не реализовано в нескольких местах независимо.
- Решение можно объяснить без длинной цепочки скрытых допущений и косвенных уровней.

### AP-018 — Ясность и одна ответственность
**Добавлено:** 2026-03-01 00:40  
**Описание:** Читаемость и локальность изменений важнее "умных" решений.  
**Почему:**
- Поддерживаемость важнее разовой изобретательности.
- Ясный код лучше переносит handoff между разработчиками и агентами.
- Локальность изменений уменьшает вероятность побочных эффектов.  
  **Требование:**
- Ясность важнее умности: код должен быть понятен без скрытого контекста.
- Модуль/файл имеет одну основную ответственность и одну причину для изменения.
- Контракт важнее реализации: внешнее поведение фиксируется тестами/контрактами.  
  **Критерии проверки:**
- Основную роль файла можно объяснить одним предложением.
- Изменение одного сценария не требует знания несвязанных деталей в том же модуле.
- Внешнее поведение модуля подтверждается тестом, примером контракта или явно описанным API.

### AP-019 — Лимиты размера и смысловое дробление
**Добавлено:** 2026-03-01 00:40  
**Описание:** Размер и сложность ограничиваются, чтобы код оставался обслуживаемым.  
**Почему:**
- Большие файлы и функции хуже ревьюятся и чаще скрывают несколько ответственностей.
- Ограничения по размеру работают как ранний сигнал к рефакторингу.
- Смысловое дробление упрощает навигацию и ownership.  
  **Требование:**
- Soft limit файла: до 150 строк.
- Hard limit файла: 250 строк; превышение требует рефактора до merge.
- Функция: до 40 строк; cyclomatic complexity: до 10.
- Файл дробится по доменным понятиям, если содержит более одной ответственности или независимые смысловые блоки.
- Формат: 1 файл = 1 ответственность + 1 главный экспорт.
- Антипаттерны запрещены: репозиторные `utils.ts`, `helpers.ts`, `common.ts`.  
  **Критерии проверки:**
- Превышение лимита сопровождается рефакторингом, а не молчаливым исключением.
- Большой файл разбит по смысловым ролям, а не по случайным техническим признакам.
- Общие модули не превращаются в свалку разнородной логики без bounded context.

### AP-020 — Clean Architecture границы и composition root
**Добавлено:** 2026-03-01 00:40  
**Описание:** Зависимости направлены внутрь, а бизнес-правила и use-case-сценарии изолированы от деталей доставки, хранения и интеграций.  
**Почему:**
- Бизнес-логика должна меняться медленнее инфраструктуры.
- Явные границы снижают связность и упрощают замену интеграций.
- Composition root делает wiring явным и контролируемым.
- Use-case слой должен владеть сценарием, а не растворяться в transport или persistence-деталях.  
  **Требование:**
- Слои определяются так:  
  Domain — сущности, value objects, доменные инварианты и доменные правила.  
  Application — use-cases/interactors, orchestration бизнес-сценария и контракты портов, через которые приложению нужны внешние зависимости.  
  Interface — HTTP/CLI/Jobs/UI adapters, которые принимают вход, вызывают use-case и маппят результат наружу.  
  Infrastructure — реализации портов и интеграции с DB, API, очередями, файлами и внешними сервисами.  
  Composition Root — место, где выбираются конкретные реализации и связываются зависимости.
- Domain не импортирует Application, Interface и Infrastructure.
- Application импортирует Domain и собственные контракты портов, но не конкретные инфраструктурные реализации.
- Use-case/interactor владеет бизнес-сценарием и координирует порты; repository/gateway не должен становиться владельцем бизнес-правил, авторизации и transport-логики.
- Interface не содержит бизнес-правил и не обращается напрямую к DB/API в обход use-case.
- Infrastructure не импортируется внутрь бизнес-слоев; исключение только wiring в composition root.  
  **Критерии проверки:**
- Use-case можно протестировать без реальной DB и внешних API через подмену портов.
- Замена адаптера не требует переписывания Domain/Application.
- Entry point не содержит бизнес-логики и не знает деталей хранения.
- Конкретный выбор клиента, репозитория, очереди или провайдера сосредоточен в composition root.
- Бизнес-команда имеет явный use-case/interactor-owner, а не размазана по route/controller/service/repository.

### AP-025 — Дисциплина импортов
**Добавлено:** 2026-03-01 00:40  
**Описание:** Граф зависимостей должен оставаться ацикличным и предсказуемым.  
**Почему:**
- Циклы размывают границы владения и делают код хрупким.
- Предсказуемый граф зависимостей упрощает рефакторинг и анализ влияния.
- Локальные import-правила позволяют автоматизировать архитектурный контроль.  
  **Требование:**
- Circular dependencies запрещены.
- Barrel exports по умолчанию запрещены; допускаются только локально и осознанно.  
  **Критерии проверки:**
- Автоматическая проверка или статический анализ не находит циклических зависимостей.
- Модуль можно импортировать без подтягивания скрытого набора несвязанных экспортов.
- Barrel используется только там, где он не размывает границы модуля.

## Границы, контракты и данные

### AP-012 — Fail-fast и явные контракты ошибок
**Добавлено:** 2026-02-12 13:30  
**Описание:** Невалидные данные и нарушения предпосылок останавливаются как можно раньше с предсказуемым результатом.  
**Почему:**
- Ошибка фиксируется ближе к источнику.
- Снижается риск скрытой порчи состояния и каскадных сбоев.
- Поведение системы остается диагностируемым и однозначным.  
  **Требование:**
- Входные данные валидируются на доверенной границе входа или на границе use-case до выполнения бизнес-правил.
- Ошибки классифицируются по типу ответственности: validation, domain, conflict, auth/access, rate-limit, infrastructure, unexpected.
- Нарушение обязательных предпосылок не маскируется fallback-поведением.
- После критической ошибки запрещено тихо продолжать выполнение как будто операция успешна.  
  **Критерии проверки:**
- Для каждого public entry point существует явная валидация входа или делегирование в общий валидатор.
- Ошибки приложения централизованно отображаются в transport-specific ответ в interface-layer.
- В коде нет silent-fail веток после нарушения инварианта.

### AP-013 — Целостность пользовательских данных (без фолбэков)
**Добавлено:** 2026-02-12 13:30  
**Описание:** Если обязательные пользовательские данные отсутствуют или повреждены, система должна явно сигнализировать об ошибке, а не подменять данные эвристиками.  
**Почему:**
- Подмена обязательных атрибутов создает скрытую порчу данных.
- Ошибка целостности должна быть заметна до записи неверного состояния.
- Совместимость не должна разрушать доверие к пользовательским данным.  
  **Требование:**
- Не подставлять альтернативные идентификаторы/значения для обязательных пользовательских атрибутов.
- Возвращать явную ошибку валидации/конфликта и логировать контекст инцидента.
- Временные совместимости допускаются только если это явно зафиксировано в feature-spec как переходный этап и не нарушает целостность данных.  
  **Критерии проверки:**
- Отсутствие обязательного поля не приводит к тихой подмене соседним значением.
- Ошибка целостности заметна в результате операции и в диагностическом контексте.
- Переходная совместимость оформлена как явное исключение, а не как скрытый permanent fallback.

### AP-017 — Минимизация данных и allowlist-экспорт
**Добавлено:** 2026-02-12 14:20  
**Описание:** Внешние контракты и логи должны раскрывать только минимально необходимый объем данных по allowlist-принципу.  
**Почему:**
- Избыточный экспорт данных повышает риск утечек и случайной связности.
- Явный allowlist делает контракт контролируемым и обозримым.
- Логи не должны становиться теневой базой чувствительных данных.  
  **Требование:**
- Для внешних ответов и событий логирования определяется явный список разрешенных полей (default deny для прочих полей).
- Чувствительные данные (PII, секреты, токены, внутренние технические детали) не выдаются и не логируются без явной необходимости.
- Добавление новых наружных полей и логируемых атрибутов должно быть явно зафиксировано в требованиях фичи.
- Маскирование/редакция чувствительных значений выполняется до записи в лог.  
  **Критерии проверки:**
- DTO и log payload собираются через явный набор полей, а не через "отдать/залогировать объект целиком".
- Новое наружное поле появляется как осознанное изменение контракта.
- Чувствительные значения маскируются до попадания в sink логирования.

### AP-021 — Тонкие entry points
**Добавлено:** 2026-03-01 00:40  
**Описание:** HTTP/CLI/Jobs выполняют только оркестрацию границы.  
**Почему:**
- Бизнес-логика должна жить в use-case, а не в транспортном слое.
- Тонкие entry points проще тестировать и поддерживать.
- Это удерживает единое поведение при появлении новых каналов вызова.  
  **Требование:**
- Поток entry point: parse/validate input -> enforce security/limits -> call use-case/workflow -> map result to output.
- Command entry point делегирует в command use-case; query entry point делегирует в query path/read model.
- В entry points запрещены бизнес-правила, прямые DB/API вызовы и сложная оркестрация.  
  **Критерии проверки:**
- Основная логика действия находится в use-case или workflow, а не в route/command/handler.
- Entry point можно читать как тонкий адаптер без знания предметной области.
- Прямой вызов DB/API из entry point отсутствует, кроме технической инфраструктуры самой границы.
- Из entry point видно, идет ли поток в command-операцию или в read/query-операцию.

### AP-022 — Typed errors и явная обработка исключений
**Добавлено:** 2026-03-01 00:40  
**Описание:** Ошибки типизированы и централизованно маппятся на публичные ответы.  
**Почему:**
- Типизированные ошибки делают поведение системы предсказуемым.
- Централизация маппинга предотвращает хаос в обработке исключений.
- Явное игнорирование ошибок уменьшает число скрытых дефектов.  
  **Требование:**
- Используется типизированная ошибка уровня приложения (`code`, `message`, `publicStatus`, `cause`) или эквивалентный структурный формат без жесткой привязки к HTTP.
- Маппинг внутренних ошибок в HTTP/CLI/job-specific публичные результаты выполняется в одном месте interface-layer.
- Empty `catch` запрещен. Допустимы только: rethrow/typed-wrap, log+continue для best-effort, convert в Result/Either.
- Намеренное игнорирование ошибки маркируется явно: `// intentionally ignored: <reason>` или `ignoreError(err, { reason })`.
- В background/async процессах запрещено молча глотать ошибки: минимум лог + контекст.  
  **Критерии проверки:**
- Обработчик ошибок можно найти в одном понятном boundary-месте.
- Исключение не исчезает бесследно без rethrow, wrap или явного best-effort решения.
- На уровне приложения есть ограниченный и понятный набор error codes/типов.

### AP-026 — API-контракты и DTO границы
**Добавлено:** 2026-03-01 00:40  
**Описание:** Публичные интерфейсы стабильны и не завязаны на форму хранения.  
**Почему:**
- Хранилище и внешний контракт эволюционируют с разной скоростью.
- Прямая утечка DB-модели наружу повышает связность и ломкость API.
- Явные DTO-границы упрощают versioning и контроль изменений.  
  **Требование:**
- Любое публичное API покрывается контрактными тестами.
- Внешний DTO не должен быть прямым отражением DB-сущности.
- Command DTO и query/read DTO проектируются под внешний контракт и не обязаны совпадать между собой.
- Нормализация/маппинг выполняются в явно выделенной boundary-зоне.  
  **Критерии проверки:**
- Изменение схемы хранения не требует автоматического изменения внешнего API.
- DTO собирается отдельным маппингом, а не сериализацией persistence-модели как есть.
- Контрактный тест фиксирует ожидаемую форму публичного ответа или запроса.
- Command payload, read payload и persistence-модель могут эволюционировать независимо без скрытой связности.

### AP-027 — Конфигурация и окружение (fail-fast)
**Добавлено:** 2026-03-01 00:40  
**Описание:** Конфигурация централизована и валидируется на старте.  
**Почему:**
- Ошибка конфигурации должна проявляться при запуске, а не в середине выполнения.
- Централизация конфигурации уменьшает скрытые зависимости между модулями.
- Это упрощает эксплуатацию и повторяемость окружений.  
  **Требование:**
- Обязательные env-переменные валидируются при запуске; при нарушении — fail fast.
- Секреты в репозитории запрещены.
- Конфигурация собирается в одном модуле/слое.  
  **Критерии проверки:**
- Приложение не стартует с неполной обязательной конфигурацией.
- Модули получают уже проверенную конфигурацию, а не читают env произвольно по месту.
- Секретные значения не зафиксированы в коде и не коммитятся в репозиторий.

## Надежность, безопасность, наблюдаемость

### AP-016 — Доверенная security-граница только на authoritative boundary
**Добавлено:** 2026-02-12 14:20  
**Описание:** Решения по авторизации и доступу считаются достоверными только на доверенной исполняющей границе системы.  
**Почему:**
- Клиент, UI и внешние вызовы контролируются недоверенной стороной.
- Одно и то же действие должно проверяться одинаково независимо от канала вызова.
- Иначе защита расползается по интерфейсам и становится обходной.  
  **Требование:**
- Проверки авторизации и доступа выполняются на authoritative execution boundary для каждого защищенного действия.
- UI- и client-side проверки допускаются только как UX-подсказка и не дают права на выполнение операции.
- Прямой вызов API, job, webhook handler или другого entry point не должен обходить те же правила доступа.
- Идентичность, claims и security context валидируются до передачи управления в бизнес-логику.  
  **Критерии проверки:**
- Каждый защищенный use-case имеет серверную или иную доверенную точку проверки доступа.
- Решение о доступе не основывается только на клиентских флагах, ролях из UI-состояния или скрытых кнопках.
- Альтернативные каналы вызова используют ту же политику доступа, что и основной интерфейс.

### AP-023 — Async дисциплина и no floating promises
**Добавлено:** 2026-03-01 00:40  
**Описание:** Асинхронный код должен быть управляемым и наблюдаемым.  
**Почему:**
- Потерянные async-ветки сложно диагностировать и тестировать.
- Осознанный fire-and-forget требует такого же контроля, как и обычный await.
- Async-дисциплина снижает долю скрытых race-condition и подвисаний.  
  **Требование:**
- Любой Promise должен быть `await` или явно обработан.
- `void` допускается только для осознанного fire-and-forget с логированием результата.
- Для best-effort async операций добавляются timeout (где уместно) и логирование success/failure.  
  **Критерии проверки:**
- В коде нет плавающих Promise без явного решения.
- Fire-and-forget операции выделены визуально и сопровождаются диагностикой результата.
- Потенциально долгие best-effort операции имеют ограничение по времени или явную стратегию завершения.

### AP-024 — Наблюдаемость и защита чувствительных данных
**Добавлено:** 2026-03-01 00:40  
**Описание:** Логи являются структурным источником диагностики без утечки секретов и с явной семантикой `severity`, `outcome` и `error_type`.  
**Почему:**
- Без наблюдаемости сложнее расследовать инциденты и деградации.
- Неструктурные логи плохо агрегируются и анализируются.
- Чувствительные данные не должны попадать в операционные артефакты.
- Severity, outcome и тип ошибки решают разные задачи и не должны смешиваться в одном поле.
- Диагностические логи должны быть пригодны для операционного дебага, а не быть случайным текстовым потоком.  
  **Требование:**
- Логи структурные (JSON).
- Для каждого request/job используется correlation id.
- Уровень логирования выбирается централизованной конфигурацией и может меняться по окружению без правки бизнес-кода.
- Каждая значимая диагностическая запись должна содержать минимум: `event`, `severity`, `outcome`, `correlation_id`, `operation` и релевантный контекст сущности/актора.
- В диагностическом логе отдельно фиксируются как минимум: `severity`, `outcome`, `error_type` и `error_code` (если применимо).
- `severity` отражает операционную срочность (`debug|info|warn|error`) и не заменяет собой `outcome` или `error_type`.
- `outcome` отражает результат операции (`success|rejected|failed` или эквивалент).
- `error_type` используется для классификации причины отказа (`validation_error`, `access_denied`, `conflict`, `dependency_error`, `timeout_error`, `db_error`, `unexpected_error` и т.п.).
- Ожидаемый бизнес-отказ не обязан иметь `severity=error`, если система работает штатно.
- Запрещено логировать OAuth codes, access/refresh tokens, пароли, секреты, сырые email (только mask/hash).
- Ключевые действия логируются с outcome, latency и контекстом.  
  **Критерии проверки:**
- По логу можно связать цепочку событий одного запроса или job.
- Операционные события содержат outcome и базовый контекст для диагностики.
- Секреты и чувствительные атрибуты не попадают в log sink в сыром виде.
- По записи лога можно отдельно понять срочность, результат операции и природу ошибки.
- Реальный системный сбой не тонет в потоке ожидаемых бизнес-отказов.

### AP-028 — Стратегия тестирования и детерминизм
**Добавлено:** 2026-03-01 00:40  
**Описание:** Тесты подтверждают поведение по слоям и остаются стабильными.  
**Почему:**
- Разные слои требуют разных видов проверки.
- Нестабильные тесты разрушают доверие к обратной связи.
- Детерминизм снижает ложные падения и упрощает диагностику.  
  **Требование:**
- Core логика покрывается unit-тестами без I/O.
- Границы DB/external APIs покрываются integration/contract тестами.
- Тесты детерминированы: контролируются время, TZ, random.  
  **Критерии проверки:**
- Unit-тесты бизнес-логики не требуют сети, реальной БД и побочных эффектов.
- Контракты интеграций проверяются отдельным типом тестов, а не только через e2e.
- Один и тот же тест не зависит от локальной TZ, текущего времени и случайных значений.

## Условные правила по стеку

### AP-029 — TypeScript strictness (если стек на TS)
**Добавлено:** 2026-03-01 00:40  
**Применимость:** только для TypeScript-проектов  
**Описание:** Типобезопасность обязательна на границах и в доменной логике.  
**Почему:**
- TS приносит пользу только при строгой дисциплине на границах.
- Runtime-валидация нужна там, где компилятор уже не контролирует внешние данные.
- Ослабление strictness быстро размывает надежность всей кодовой базы.  
  **Требование:**
- `any` запрещен либо строго ограничен с явным обоснованием.
- Non-null assertion `!` запрещен либо строго ограничен с явным обоснованием.
- Внешние данные валидируются runtime-схемой (zod/valibot/io-ts или эквивалент).  
  **Критерии проверки:**
- Использование `any` и `!` единично, локально и сопровождается объяснимой причиной.
- Границы внешнего ввода не полагаются только на compile-time типы.
- Наружные данные проходят через runtime-проверку до входа в доменную логику.

### AP-063 — Базовый frontend stack для веб-приложений

- Единый stack baseline уменьшает случайную архитектурную вариативность и стоимость поддержки.
- Понятный дефолт ускоряет разработку, code review и onboarding.
- Это снижает риск "каждый экран на своей любимой библиотеке".  
  **Требование:**

- Формы по умолчанию строятся на React Hook Form.
- Runtime validation на клиенте и на общих схемах по умолчанию строится на Zod.
- Zustand используется только для небольшого global UI/app state и не заменяет собой server state layer.  
  **Критерии проверки:**
- Новый frontend-проект не тащит альтернативный stack без явной технической причины.
- Server state, form state и small global state используют разные инструменты по назначению.
- В проекте нет конкурирующих дефолтных библиотек одного класса без обоснования.

### AP-066 — TypeScript как обязательный язык frontend и Node.js backend
**Добавлено:** 2026-03-08 00:40  
**Применимость:** для frontend и backend на Node.js  
**Описание:** TypeScript является обязательным языком по умолчанию для frontend-кода и backend-кода на Node.js.  
**Почему:**
- Единая типовая модель между frontend и Node.js backend снижает интеграционные ошибки.
- TypeScript делает контракты и refactor-safe изменения дешевле в long-lived codebase.
- Это упрощает shared schemas, DTO и boundary validation.  
  **Требование:**
- Frontend код пишется на TypeScript.
- Если backend работает на Node.js, backend код пишется на TypeScript.
- JavaScript допускается только как явное исключение для tooling/bootstrap/legacy-edge cases.
- Shared contracts между frontend и Node.js backend по возможности выражаются типами и runtime-валидацией, а не только неформальными договоренностями.  
  **Критерии проверки:**
- Новый application code на frontend и Node.js backend не пишется на plain JavaScript без причины.
- Shared boundary-контракты не существуют только "в голове команды".
- Исключения из TS-правила локальны и объяснимы.

- dotenv не обязателен там, где framework уже предоставляет корректную env lifecycle model.
- Env loading не отменяет fail-fast конфигурацию и не является заменой явной схемы env-переменных.
- Отклонение от Drizzle baseline требует явного технического обоснования.  
  **Критерии проверки:**
- Для нового Node.js backend не появляется случайный ORM/config-loader без причины.
- Env bootstrap происходит единообразно и не размазан хаотично по месту.
- Drizzle/data layer интегрирован с общими правилами транзакций, миграций и boundary-моделей.

## Дополнительные baseline-правила для command-side consistency

### AP-032 — Command/Query Separation
**Добавлено:** 2026-03-07 22:10  
**Описание:** Пути изменения состояния и пути чтения проектируются раздельно и не смешивают свои обязанности.  
**Почему:**
- Командные операции требуют консистентности, атомарности и контроля конкурентности.
- Чтение чаще оптимизируется под форму ответа, фильтрацию и производительность.
- Смешение command и query в одном слое быстро размывает инварианты и ответственность.  
  **Требование:**
- Command-путь отвечает только за изменение состояния и соблюдение инвариантов.
- Query-путь отвечает только за чтение и построение read-model/response-model.
- Query-операция не должна неявно изменять состояние системы.
- Command-операция не должна тащить в себя произвольную read-логику, не связанную с обеспечением инварианта.  
  **Критерии проверки:**
- Для state-changing сценария существует явный command use-case/handler/interactor.
- Для чтения существует отдельный query path/read adapter/read model.
- Из API и кода понятно, какой вызов изменяет состояние, а какой только читает.

### AP-033 — Целостность данных и атомарность команд на уровне БД
**Добавлено:** 2026-03-07 22:10  
**Описание:** Целостность и атомарность команд приоритетно обеспечиваются возможностями базы данных, а не договоренностями в коде.  
**Почему:**
- База данных является authoritative-границей состояния.
- Кодовые проверки без DB-гарантий не защищают от гонок и параллельных записей.
- Один атомарный шаг в БД надежнее серии разрозненных запросов, собранных в приложении.  
  **Требование:**
- Целостность данных обеспечивается прежде всего через constraints, unique indexes, foreign keys, check constraints, atomic statements и explicit DB transactions.
- Один бизнес-командный шаг должен стремиться к одной атомарной DB-операции; если этого недостаточно, используется одна явно оформленная DB-транзакция.
- Compensating logic в коде не используется как механизм обеспечения согласованности записей в БД.
- Паттерн `read -> check in code -> write` запрещен для command-side операций, если тот же инвариант можно выразить средствами БД (`UPSERT`, `UPDATE ... WHERE`, CTE, stored procedure, constraint, transaction).  
  **Критерии проверки:**
- Для критичного инварианта можно указать конкретную DB-гарантию, которая его защищает.
- Командная запись сводится к одной атомарной операции или к одной явной транзакционной границе.
- Ревью может показать, где именно БД, а не только код, гарантирует корректность результата.

### AP-034 — Явная стратегия конкурентного доступа
**Добавлено:** 2026-03-07 22:10  
**Описание:** Для конкурентных командных операций стратегия защиты от гонок выбирается явно, а не подразумевается неформально.  
**Почему:**
- Конкурентные изменения состояния ломают данные тише всего и дороже всего в исправлении.
- Разные инварианты требуют разных механизмов: constraint, optimistic concurrency, isolation, locking.
- Неявная стратегия почти всегда означает отсутствие реальной защиты.  
  **Требование:**
- Для каждой важной command-side операции с риском параллельного изменения выбирается явная стратегия конкурентного доступа.
- Предпочтение отдается DB constraints, optimistic concurrency и транзакционным гарантиям; pessimistic locking применяется только для коротких критических секций, когда без него нельзя удержать инвариант.
- Отсутствие явной стратегии конкурентного доступа для критичной операции считается архитектурным дефектом.
- Конкурентный доступ не должен приводить к silent overwrite, дублированию сущностей или разрушению инварианта.  
  **Критерии проверки:**
- Для рискованной записи можно назвать конкретный механизм защиты от гонки.
- Повторное конкурентное выполнение не приводит к незаметной потере данных.
- Если используется lock, его граница минимальна и ограничена по времени.

### AP-035 — Идемпотентность внешне повторяемых команд
**Добавлено:** 2026-03-07 22:10  
**Описание:** Команды, которые могут быть повторены клиентом, прокси, воркером или webhook-источником, должны быть безопасны к повторному исполнению.  
**Почему:**
- Ретраи неизбежны в сетевых и фоновых сценариях.
- Без идемпотентности одинаковая команда легко создает дубли и повторные side effects.
- Идемпотентность нужна для масштабируемой и предсказуемой обработки внешних вызовов.  
  **Требование:**
- Для externally retried commands используется idempotency key, natural unique key или эквивалентная persisted deduplication-стратегия.
- Повтор одной и той же команды должен возвращать тот же результат, безопасный already-done outcome или явный недублирующий конфликт.
- Идемпотентность обеспечивается persisted-механизмом на стороне БД/хранилища, а не временным состоянием процесса или памяти приложения.  
  **Критерии проверки:**
- Повторный submit команды не создает дублирующую запись или side effect.
- Deduplication переживает рестарт процесса и параллельное выполнение.
- Механизм идемпотентности можно указать в коде и в persistence-модели.

### AP-037 — Audit log для security и значимых бизнес-действий
**Добавлено:** 2026-03-07 22:35  
**Описание:** Security-события и значимые бизнес-действия пишутся в отдельный audit log, не смешанный с обычной диагностикой.  
**Почему:**
- Аудит отвечает на вопрос "кто и что сделал", а не "что сломалось".
- Диагностические логи шумят и могут быть неполными для разбирательства значимых действий.
- Разделение audit и diagnostic log упрощает расследование, контроль доступа и соответствие требованиям.  
  **Требование:**
- Audit log ведется отдельно от диагностических логов.
- В audit log попадают минимум: события аутентификации, смена роли/прав, удаление, смена статуса, подтверждение/отмена значимых сущностей и другие внешне значимые действия.
- Запись audit log содержит как минимум: `audit_event`, `actor_id`, `action`, `target_type`, `target_id`, `result`, `correlation_id`, `occurred_at`.
- Audit log не используется как основной канал низкоуровневой отладки.  
  **Критерии проверки:**
- По audit log можно восстановить факт, автора и результат значимого действия без чтения обычных debug/error логов.
- Значимые security/business события не теряются среди технического шума.
- Формат audit log стабилен и не зависит от конкретного transport layer.

### AP-038 — Сквозной контекст запроса и async job traceability
**Добавлено:** 2026-03-07 22:35  
**Описание:** Контекст запроса или job должен проходить через всю цепочку выполнения, включая фоновые процессы и асинхронные переходы.  
**Почему:**
- Без сквозного контекста расследование упирается в разрозненные логи без связки.
- Асинхронные job-ы особенно быстро теряют происхождение вызова без явной передачи контекста.
- Traceability важнее количества логов: нужно уметь связать события в историю.  
  **Требование:**
- Каждый entry point создает или принимает `correlation_id/request_id` и передает его дальше по цепочке.
- При постановке async job, webhook processing, background task или retry контекст корреляции передается и логируется на принимающей стороне.
- Логи дочернего async-процесса должны сохранять связь с исходным запросом или с порождающей операцией.
- Потеря correlation context на межслойном или межпроцессном переходе считается дефектом наблюдаемости.  
  **Критерии проверки:**
- По `correlation_id` можно собрать историю запроса от entry point до job/worker и обратно к итоговому результату.
- Async-задача логирует не только собственный id, но и происхождение.
- Ретрай и фоновая обработка остаются связаны с исходной причиной запуска.

## Безопасность и злоупотребления

### AP-039 — Auth boundary и доверенная идентичность
**Добавлено:** 2026-03-07 22:55  
**Описание:** Идентичность пользователя и security context считаются достоверными только после проверки на доверенной серверной границе.  
**Почему:**
- Клиентский state, claims из UI и произвольные заголовки не являются источником истины.
- Ошибка в auth boundary быстро превращается в системную уязвимость.
- Единая точка проверки идентичности делает поведение предсказуемым для всех transport-каналов.  
  **Требование:**
- Идентичность пользователя, сессия, токен или webhook signature валидируются на доверенной boundary до входа в бизнес-логику.
- Security context передается дальше в use-case как уже проверенный и минимально достаточный набор данных.
- Клиент не определяет сам свои роли, tenant, permissions или trusted claims без серверной валидации.
- Истечение, отзыв и замена auth-артефактов должны обрабатываться как часть серверной модели безопасности, а не как UI-конвенция.  
  **Критерии проверки:**
- Защищенный entry point не передает в use-case непроверенную идентичность.
- Источник роли/tenant/access claims можно указать на серверной стороне.
- Auth boundary одинакова по смыслу для HTTP, background triggers и webhook-ов.

### AP-040 — Авторизация на каждом защищенном use-case
**Добавлено:** 2026-03-07 22:55  
**Описание:** Каждый защищенный сценарий проверяет право на выполнение действия в явной доверенной точке.  
**Почему:**
- Проверка только на уровне UI или route не гарантирует защиту для альтернативных каналов вызова.
- Авторизация должна жить рядом с бизнес-действием, а не зависеть от того, как до него дошли.
- Явная проверка прав упрощает ревью и снижает риск обхода.  
  **Требование:**
- Каждый protected command/query/use-case имеет явную серверную проверку авторизации.
- Проверка авторизации учитывает не только роль, но и тип действия над конкретным ресурсом, если это требуется моделью доступа.
- UI-скрытие кнопок, disabled controls и client-side guards не считаются достаточной защитой.
- Отсутствие явной проверки доступа для защищенного сценария считается архитектурным дефектом.  
  **Критерии проверки:**
- Для каждого защищенного use-case можно показать, где и по каким данным принимается access decision.
- Прямой вызов API/handler вне UI не обходит проверку доступа.
- На ревью понятно, почему данному actor разрешено или запрещено действие.

### AP-041 — Object-level access и tenant isolation
**Добавлено:** 2026-03-07 22:55  
**Описание:** Доступ к конкретному объекту и данным арендатора проверяется явно и не основывается только на знании идентификатора.  
**Почему:**
- Большинство практических утечек данных в веб-приложениях происходят на object-level access, а не на "общем отсутствии auth".
- Multi-tenant и owner-scoped данные требуют отдельной дисциплины изоляции.
- Простая проверка "пользователь залогинен" не защищает от доступа к чужой записи.  
  **Требование:**
- Для owner-scoped и tenant-scoped ресурсов выполняется явная проверка принадлежности объекта перед чтением или изменением.
- Знание `id`, `slug` или другого внешнего идентификатора само по себе не дает доступа к ресурсу.
- Tenant boundary применяется одинаково для query и command операций.
- Кросс-tenant чтение или запись допускаются только для явно авторизованных административных сценариев.  
  **Критерии проверки:**
- Нельзя получить или изменить чужой объект только через подбор/знание идентификатора.
- У каждого tenant-scoped запроса можно указать, где применяется tenant filter или equivalent guard.
- Административные обходы tenant boundary явно видны и ограничены.

### AP-042 — Секреты и доверенная конфигурация
**Добавлено:** 2026-03-07 22:55  
**Описание:** Секреты и security-sensitive параметры управляются как доверенная конфигурация, а не как данные кода или клиента.  
**Почему:**
- Утечка секрета часто происходит раньше, чем эксплуатация любой другой уязвимости.
- Разрозненное управление секретами разрушает воспроизводимость окружений и аудит изменений.
- Security-sensitive конфигурация должна иметь такой же fail-fast и ownership, как и бизнес-критичные env.  
  **Требование:**
- Секреты, signing keys, API keys, DB credentials и токены не хранятся в коде, тестовых фикстурах или клиентском бандле.
- Security-sensitive конфигурация читается только из доверенного конфигурационного слоя и валидируется на старте.
- Секреты не попадают в логи, ошибки, DTO, debug dumps и audit log.
- Ротация и замена секретов не должны требовать переписывания бизнес-кода.  
  **Критерии проверки:**
- В репозитории нет hardcoded credentials и секретов в обычных конфигурационных файлах.
- Security-sensitive env проверяются централизованно и fail-fast.
- При диагностике система не раскрывает secret values в явном виде.

### AP-043 — Безопасная обработка входа, файлов и webhook-ов
**Добавлено:** 2026-03-07 22:55  
**Описание:** Любой внешний ввод, включая body, query, headers, файлы и webhook payloads, проходит проверку и ограничения до использования.  
**Почему:**
- Граница входа является основной поверхностью атаки.
- Файлы и webhook-ы несут дополнительные риски: spoofing, oversized payloads, unexpected formats.
- Runtime-валидация и ограничения размера предотвращают целый класс дефектов до бизнес-логики.  
  **Требование:**
- Внешний ввод валидируется runtime-схемой или эквивалентным явным валидатором до использования.
- Mass assignment и неявное маппирование "все поля запроса в модель" запрещены.
- Для upload/download задаются явные ограничения по размеру, допустимому типу и правилам обработки.
- Webhook-ы принимаются только после проверки подписи, происхождения или эквивалентного механизма доверия.  
  **Критерии проверки:**
- Наружные payloads не попадают в доменную логику без валидации.
- Файловая обработка не принимает произвольные размеры и типы "по умолчанию".
- Webhook handler может показать, где именно выполняется trust check до обработки.

### AP-044 — Безопасные исходящие вызовы и внешние интеграции
**Добавлено:** 2026-03-07 22:55  
**Описание:** Исходящие сетевые вызовы выполняются через контролируемые адаптеры с ограничениями доверия, времени и адресатов.  
**Почему:**
- Внешние вызовы расширяют attack surface не меньше, чем входящие.
- Бесконтрольные URL и сетевые запросы создают SSRF, data exfiltration и нестабильность.
- Controlled outbound policy делает поведение интеграций предсказуемым и ревьюемым.  
  **Требование:**
- Внешние HTTP/API вызовы выполняются через явные adapters/clients, а не через произвольные ad-hoc вызовы по коду.
- Для исходящих запросов задаются timeout, retry policy и ограничения доверия к адресатам там, где это применимо.
- Пользовательский ввод не должен напрямую определять произвольный target URL без allowlist или эквивалентной защиты.
- Outbound integration не должна раскрывать лишние данные внешней стороне сверх явно заданного контракта.  
  **Критерии проверки:**
- В коде можно найти централизованный клиент или adapter для каждой внешней интеграции.
- Пользовательский payload не позволяет незаметно инициировать запрос к произвольному внутреннему/внешнему адресу.
- Сбой внешней зависимости диагностируется и ограничен по времени.

### AP-045 — Защита от злоупотреблений и resource exhaustion
**Добавлено:** 2026-03-07 22:55  
**Описание:** Публичные и дорогие операции защищаются от brute-force, replay и чрезмерного расхода ресурсов.  
**Почему:**
- Даже корректная бизнес-логика может быть выведена из строя через abuse, а не через "взлом".
- Auth endpoints, expensive queries и side-effect commands требуют отдельных ограничений.
- Базовые лимиты дешевле инцидента и лучше масштабируются, чем ручная реакция.  
  **Требование:**
- Для auth endpoints, externally retried commands и дорогостоящих операций применяются rate limits, backoff, replay protection или эквивалентные защитные механизмы.
- Размер входного payload, число параллельных операций и стоимость тяжелых запросов должны быть ограничены.
- Повторная попытка и злоупотребление не должны приводить к неограниченному числу expensive side effects.
- Security- и abuse-relevant события логируются так, чтобы их можно было расследовать отдельно от обычной бизнес-диагностики.  
  **Критерии проверки:**
- Система не позволяет бесконечно долбить login/reset/expensive command без защитного механизма.
- Тяжелый endpoint имеет явное ограничение или защиту от ресурсного истощения.
- Replay и brute-force сценарии учитываются в модели обработки внешних вызовов.

## Данные и интеграции

### AP-046 — Эволюция контрактов и схем без ломающих изменений по умолчанию
**Добавлено:** 2026-03-07 23:10  
**Описание:** Публичные контракты и схема хранения развиваются по совместимой стратегии, если не согласован явный breaking change.  
**Почему:**
- API, webhook-и, read models и БД почти никогда не обновляются одновременно.
- Ломающие изменения в данных и контрактах часто проявляются уже после деплоя и дороже исправляются, чем предотвращаются.
- Совместимая эволюция упрощает rollout, миграции и откат.  
  **Требование:**
- Изменения публичных контрактов и логической модели данных делаются по принципу expand -> migrate -> contract, а не через одномоментную замену старого новым.
- Удаление или переименование публичного поля, события, колонки или статуса без этапа совместимости допускается только как явно согласованный breaking change.
- Новая схема или контракт должны быть пригодны к сосуществованию со старой версией на период rollout/migration.
- Переход логической модели данных не должен предполагать одномоментное атомарное обновление всех читателей и писателей системы.  
  **Критерии проверки:**
- Для потенциально ломающего изменения можно показать промежуточный совместимый шаг.
- Изменение схемы БД не требует атомарного одновременного обновления всех читающих и пишущих компонентов.
- На ревью видно, как система переживает rollout старой и новой версии одновременно.

### AP-047 — Надежные side effects после изменения состояния
**Добавлено:** 2026-03-07 23:10  
**Описание:** Внешние side effects после записи в БД выполняются через надежный persisted boundary, а не через хрупкую пост-командную импровизацию.  
**Почему:**
- Классический сбой: запись в БД прошла, а событие, письмо, webhook или задача в очередь потерялись.
- Простое "сначала сохранить, потом отправить" без persisted coordination дает рассинхрон между состоянием системы и внешним миром.
- Надежный persisted boundary упрощает ретраи, дедупликацию и расследование сбоев.  
  **Требование:**
- Если после успешной командной записи требуется внешний side effect, его постановка должна быть связана с изменением состояния через persisted-механизм (например, outbox или эквивалент).
- Внешний side effect не должен зависеть только от временного состояния процесса после commit.
- Повторная доставка и ретраи side effect должны быть совместимы с идемпотентностью получателя или защищены дедупликацией.
- Потеря side effect между DB commit и внешней отправкой считается архитектурным дефектом.  
  **Критерии проверки:**
- Для важного внешнего эффекта можно показать persisted-след того, что он должен быть доставлен.
- Перезапуск процесса не теряет уже созданный, но еще не доставленный side effect.
- Отправка во внешний мир не опирается на "надеемся, что код после commit успеет выполниться".

### AP-048 — Явное соответствие внутренних и внешних идентификаторов
**Добавлено:** 2026-03-07 23:10  
**Описание:** Интеграции с внешними системами используют явное и устойчивое хранение соответствия внутренних и внешних идентификаторов.  
**Почему:**
- Поиск "по email", "по имени" или другим mutable полям быстро становится источником дублей и рассинхрона.
- Внешние системы редко гарантируют ту же модель идентичности, что и внутренняя БД.
- Явный mapping упрощает идемпотентность, reconcilliation и повторную синхронизацию.  
  **Требование:**
- Для внешних интеграций хранится явное соответствие internal id <-> external id или эквивалентный устойчивый linkage-механизм.
- Mutable business attributes не используются как единственный ключ синхронизации, если можно хранить стабильный mapping.
- Правила создания, обновления и повторного связывания external identity должны быть детерминированы.
- Потеря связи между внутренней и внешней сущностью должна быть обнаружима и диагностируема.  
  **Критерии проверки:**
- Для сущности, синхронизируемой наружу, можно явно показать источник external id.
- Повторная синхронизация не создает дубли только из-за изменения имени/email/label.
- При сбое интеграции можно восстановить связь сущностей без ручной археологии по логам.

### AP-049 — Явная семантика удаления, архивации и историчности
**Добавлено:** 2026-03-07 23:10  
**Описание:** Для каждой значимой сущности явно определено, что означает delete, archive, deactivate или anonymize, и как это влияет на связи и интеграции.  
**Почему:**
- Неявная семантика удаления ломает аудит, интеграции и восстановление состояния.
- Hard delete по умолчанию опасен для сущностей, на которые ссылаются внешние системы, логи или исторические процессы.
- Явная политика жизненного цикла данных делает поведение предсказуемым для кода и эксплуатации.  
  **Требование:**
- Для значимых сущностей определяется допустимая модель: hard delete, soft delete, archive, deactivate, anonymize или их сочетание.
- Удаление не должно тихо разрушать ссылки, аудит или внешние интеграции.
- Read/query path явно определяет, видит ли он archived/deleted/inactive записи и при каких условиях.
- Если сущность имеет юридическую, финансовую, audit- или integration-значимость, hard delete по умолчанию не считается безопасным выбором.  
  **Критерии проверки:**
- Для важной сущности можно ответить, что именно происходит при "удалении".
- Удаленная или архивная запись не превращается в неконсистентную дыру для связанных данных.
- Поведение read-side относительно archived/deleted записей явно определено.

### AP-050 — Справочные данные, статусы и перечисления как управляемые контракты
**Добавлено:** 2026-03-07 23:10  
**Описание:** Статусы, enum-значения и справочные данные управляются как явные контракты, а не как стихийный набор строк по коду и интеграциям.  
**Почему:**
- Статусы и справочники быстро расползаются между БД, кодом, UI и внешними системами.
- "Магические строки" ломают миграции, аналитику и интеграции незаметнее, чем ошибки типов.
- Управляемый статусный словарь упрощает переходы состояний и версионность контрактов.  
  **Требование:**
- Значимые статусы, типы и reference data определяются централизованно или через явный source of truth.
- Недопустимы независимые несогласованные наборы status/enum значений в разных слоях для одной и той же бизнес-сущности.
- Изменение статуса или reference value рассматривается как контрактное изменение, а не как локальная строковая правка.
- Переходы между статусами, если они важны для бизнеса, валидируются явно, а не допускаются произвольно.  
  **Критерии проверки:**
- Для ключевой сущности можно показать канонический список статусов/типов и допустимые переходы.
- Новый статус не требует искать и чинить десятки магических строк вручную.
- Интеграции, БД и read models используют согласованное множество reference values.

## Frontend и клиентское приложение

### AP-051 — Frontend feature boundaries и separation of concerns
**Добавлено:** 2026-03-08 00:20  
**Описание:** Клиентское приложение строится по feature-границам и не смешивает UI, бизнес-правила, transport-адаптацию и случайный shared-код в одном слое.  
**Почему:**
- Фронтенд деградирует не от отсутствия компонентов, а от размытия границ ответственности.
- Feature boundary делает изменение экрана и сценария локальным, а не глобальным.
- Shared-слой должен обслуживать повторяемость, а не быть свалкой случайной логики.  
  **Требование:**
- Код клиентского приложения группируется вокруг feature/user-flow, а не только вокруг технических папок.
- UI-примитивы, data access, orchestration и feature-specific state не смешиваются без явной причины.
- Shared/frontend-common слой содержит только действительно переиспользуемые примитивы и контракты.
- Бизнес-правила, завязанные на backend contract или auth semantics, не размазываются по произвольным компонентам.  
  **Критерии проверки:**
- Основной пользовательский сценарий можно локализовать в одном feature slice.
- Изменение одной feature не требует правки большого количества unrelated shared-файлов.
- Shared-компоненты не знают доменную специфику отдельных feature.

### AP-052 — Явное разделение client state, server state и form state
**Добавлено:** 2026-03-08 00:20  
**Описание:** Состояние интерфейса, серверные данные и состояние формы управляются разными механизмами и не сливаются в один неразборчивый store.  
**Почему:**
- У этих типов state разные источники истины, жизненный цикл и стоимость ошибки.
- Смешение server state и UI state разрушает предсказуемость кэша, invalidation и optimistic updates.
- Явное разделение упрощает дебаг, тестирование и performance-оптимизацию.  
  **Требование:**
- Server state кэшируется и обновляется как server state, а не как произвольная локальная копия "на всякий случай".
- UI state и transient state не используются как источник истины для данных, принадлежащих серверу.
- Form state управляется отдельно от кэша запросов и отдельно от глобального UI state.
- Мутации и invalidation выполняются через явный mutation flow, а не через случайные локальные патчи по дереву компонентов.  
  **Критерии проверки:**
- Для каждого типа state можно указать его source of truth.
- Обновление server state не требует ручного синхрона десятков локальных копий.
- Форма, кэш запросов и UI-флаги не завязаны в один скрытый узел зависимостей.

### AP-053 — React component boundaries и rendering discipline
**Добавлено:** 2026-03-08 00:20  
**Применимость:** только для React-проектов  
**Описание:** React-компоненты и hooks строятся так, чтобы рендер был предсказуемым, а побочные эффекты были локализованы.  
**Почему:**
- React-приложение деградирует от неявных side effects и хаотичного состояния быстрее, чем от нехватки абстракций.
- Предсказуемый рендер и чистые границы hooks упрощают debugging и performance-tuning.
- Локализация side effects уменьшает accidental re-renders и race conditions на клиенте.  
  **Требование:**
- Компоненты по умолчанию остаются чистыми относительно входных props/state; побочные эффекты изолируются в hooks или boundary-адаптерах.
- Hook не должен одновременно быть transport-клиентом, state-machine, form-engine и UI-composer без явной причины.
- Компонентные границы должны уменьшать re-render blast radius, а не расширять его.
- Derived state не дублируется без необходимости и не хранится отдельно от своего реального источника.  
  **Критерии проверки:**
- Причину re-render и источник side effect можно локализовать без чтения половины feature.
- Hooks имеют понятную роль и не превращаются в "супер-комбайн" всей страницы.
- Изменение локального UI state не дергает ненужные поддеревья без объяснимой причины.

### AP-054 — Устойчивые UI-состояния, доступность и деградация интерфейса
**Добавлено:** 2026-03-08 00:20  
**Описание:** Интерфейс явно обрабатывает loading, empty, error, denied и degraded states и остается usable при неполной доступности данных.  
**Почему:**
- Пользователь видит не "архитектуру", а то, как экран ведет себя в пограничных состояниях.
- Хорошая деградация интерфейса снижает операционную цену частичных сбоев.
- Accessibility и явные UI states уменьшают число скрытых продуктовых дефектов.  
  **Требование:**
- Для значимых экранов и flows определены loading, empty, error и access-denied состояния.
- Ошибка одного виджета или data-block не должна по умолчанию валить весь экран, если это не требуется сценарием.
- Базовая keyboard accessibility, focus management и семантика интерактивных элементов не считаются optional polish.
- Пользовательский интерфейс явно различает "нет данных", "нет доступа", "идет загрузка" и "произошел сбой".  
  **Критерии проверки:**
- У экрана есть осмысленное поведение в success и non-success состояниях.
- Частичная деградация backend не превращает весь UI в белый экран без контекста.
- Интерактивные элементы остаются доступны и понятны не только мышью.

### AP-055 — Frontend performance: caching, lazy loading и hot paths
**Добавлено:** 2026-03-08 00:20  
**Описание:** Производительность клиентского приложения проектируется вокруг hot paths, кэша и стоимости рендера, а не оставляется на финальный "оптимизационный спринт".  
**Почему:**
- Производительность деградирует постепенно и незаметно, если нет явных правил по кэшу и split-политике.
- Hot paths определяют пользовательское восприятие сильнее, чем средняя скорость редких экранов.
- Lazy loading и разумный cache policy дешевле, чем последующий heroic performance rewrite.  
  **Требование:**
- Для hot paths определяются явные performance-sensitive зоны: первый экран, авторизация, ключевые списки, формы, критичные mutations.
- Lazy loading применяется для route-level или feature-level чанков, которые не нужны на первом paint.
- Кэширование client/server state имеет явную политику freshness/invalidation и не сводится к "данные где-то лежат в памяти".
- Frontend не должен создавать render waterfalls и каскадные повторные запросы без необходимости.  
  **Критерии проверки:**
- Для ключевого user flow можно назвать основные hot paths и их основные источники стоимости.
- Тяжелые feature не грузятся в initial path без причины.
- Повторный заход на экран не провоцирует лишний шторм запросов или needless full rerender.

### AP-068 — Дизайн-система как token-driven foundation
**Добавлено:** 2026-03-08 00:55  
**Применимость:** для admin и public-facing интерфейсов  
**Описание:** Дизайн-система задает не только набор компонентов, но и единый token layer для визуальных и интерактивных решений.  
**Почему:**
- Компоненты без токенов быстро расходятся по цветам, отступам, типографике и состояниям.
- Token-driven подход делает UI консистентным и упрощает масштабирование дизайн-системы.
- Это снижает цену кастомизации и уменьшает число ad-hoc CSS-решений.  
  **Требование:**
- Дизайн-система имеет явные токены минимум для color, spacing, typography, radius, shadow, z-index и motion.
- Компоненты и variants строятся поверх токенов, а не через случайные локальные значения "по месту".
- Новые UI-решения не должны вводить второй неформальный слой визуальных констант рядом с дизайн-системой.
- Если admin и public-facing интерфейсы различаются, это делается через явные темы или variant-слой поверх общего token discipline, а не через хаотичный ad-hoc стиль.  
  **Критерии проверки:**
- В проекте можно указать source of truth для UI tokens.
- Новый компонент не тащит случайный набор hardcoded visual values без причины.
- Изменение базового визуального решения можно проводить через token/variant layer, а не через точечную правку десятков компонентов.

### AP-069 — Базовая доступность интерактивных интерфейсов
**Добавлено:** 2026-03-08 00:55  
**Применимость:** для admin и public-facing интерфейсов  
**Описание:** Базовая доступность интерфейса считается частью качества продукта, а не дополнительным polish-слоем.  
**Почему:**
- Доступность влияет не только на screen readers, но и на keyboard use, error handling, focus discipline и общую ясность интерфейса.
- Большинство практических accessibility-дефектов появляются из-за игнорирования базовых правил, а не из-за отсутствия идеального compliance-плана.
- Один общий baseline выгоднее, чем разные "послабления" для admin и public UI.  
  **Требование:**
- Интерактивные элементы должны быть доступны с клавиатуры и иметь видимый focus state.
- Семантический HTML используется по умолчанию; ARIA добавляется как дополнение, а не как замена отсутствующей семантики.
- Placeholder не заменяет label для полей ввода.
- Иконка без видимого текста допускается только если у элемента есть явный accessible name (`aria-label` или эквивалент).
- Состояния disabled, loading, error и success должны быть понятны не только визуально, но и по структуре/тексту интерфейса.  
  **Критерии проверки:**
- Основной пользовательский путь можно пройти только клавиатурой без потери управляемости.
- У интерактивных элементов есть понятный accessible name и предсказуемое поведение фокуса.
- Ошибка формы или действия не выражена только цветом, и placeholder не является единственным источником смысла.

### AP-070 — Уважение к motion preferences и предсказуемая анимация
**Добавлено:** 2026-03-08 00:55  
**Применимость:** для admin и public-facing интерфейсов  
**Описание:** Анимации и переходы не должны мешать восприятию интерфейса и обязаны уважать пользовательские настройки снижения движения.  
**Почему:**
- Motion влияет на производительность, восприятие и accessibility одновременно.
- Избыточная анимация ухудшает usability чаще, чем улучшает её.
- Поддержка reduced motion — это базовая инженерная дисциплина, а не дизайнерская опция.  
  **Требование:**
- Значимые анимации и transitions строятся осознанно и не являются обязательными для понимания интерфейса.
- Интерфейс уважает `prefers-reduced-motion` или эквивалентную пользовательскую настройку.
- Motion tokens и timings задаются централизованно, а не распределяются случайными числами по компонентам.
- Автозапуск длинных, отвлекающих или бесконечных анимаций без продуктовой причины не допускается.  
  **Критерии проверки:**
- При reduced motion интерфейс остается понятным и функциональным без потери критичной информации.
- Анимации не являются единственным способом показать изменение состояния.
- Временные параметры motion можно найти в общем token layer, а не только в локальных стилях.

## Тестирование и качество

### AP-056 — Риск-ориентированная стратегия тестирования по слоям
**Добавлено:** 2026-03-08 00:20  
**Описание:** Тесты распределяются по слоям и рискам, а не дублируют один и тот же сценарий вслепую на всех уровнях.  
**Почему:**
- Одинаковая логика, проверенная в пяти слоях, не даёт в пять раз больше доверия.
- Риск-ориентированный подход уменьшает стоимость сопровождения тестов и keeps signal high.
- Критичные сценарии должны быть покрыты, но без тестовой катастрофизации.  
  **Требование:**
- Бизнес-инварианты проверяются там, где они живут: core/use-case/integration boundary.
- E2E используются для ключевых пользовательских путей и интеграционных швов, а не как замена всем остальным тестам.
- Контрактные и integration tests покрывают boundary-точки, где чаще всего ломается совместимость.
- Для каждого важного риска можно показать, каким типом теста он покрывается и почему именно этим.  
  **Критерии проверки:**
- Нельзя увидеть слепое дублирование одного сценария во всех слоях без дополнительной ценности.
- Критичный путь пользователя имеет достаточное, но не избыточное покрытие.
- Тестовый набор можно объяснить через риск-модель, а не через "добавили еще один тест на всякий случай".

### AP-057 — Детерминированные тестовые данные и контроль flaky behavior
**Добавлено:** 2026-03-08 00:20  
**Описание:** Тестовая среда, данные и асинхронность контролируются так, чтобы сбои тестов были сигналом, а не шумом.  
**Почему:**
- Flaky tests разрушают доверие к CI быстрее, чем отсутствие части тестов.
- Нестабильные данные, время и случайность делают расследование дорогим и бесполезным.
- QA baseline должен уменьшать шум, а не производить драму вокруг каждого падения.  
  **Требование:**
- Тестовые данные создаются детерминированно и изолированно для сценария.
- Время, random, network mocks и async completion контролируются там, где это влияет на стабильность результата.
- Flaky behavior рассматривается как дефект тестовой системы и устраняется, а не нормализуется как "иногда CI красный".
- Smoke/regression тесты остаются короткими и сигнализирующими, без превращения в неуправляемый монолит.  
  **Критерии проверки:**
- Один и тот же тест стабильно воспроизводим локально и в CI.
- Падение теста можно расследовать по фиксированным данным и контролируемому окружению.
- Набор smoke/regression не деградирует в медленную и шумную свалку.

## Эксплуатация и rollout

### AP-058 — Feature flags как управляемая граница rollout
**Добавлено:** 2026-03-08 00:20  
**Описание:** Feature flags используются как управляемый механизм rollout и safety switch, а не как бессрочный слой скрытой архитектуры.  
**Почему:**
- Фича-флаги полезны для безопасного rollout, но быстро превращаются в скрытый источник сложности.
- Явный ownership и срок жизни флага уменьшают риск мертвого кода и непредсказуемых сочетаний состояний.
- Runtime kill switch дешевле аварийного хотфикса.  
  **Требование:**
- Каждый feature flag имеет owner, цель и ожидаемый срок жизни.
- Флаг не используется как постоянная замена архитектурному решению или конфигурационному хаосу.
- Для рискованных изменений должен существовать способ быстро отключить новое поведение без redeploy, если это оправдано сценарием.
- Взаимодействие нескольких флагов не должно создавать неограниченное количество непроверяемых режимов системы.  
  **Критерии проверки:**
- Для флага можно ответить, кто за него отвечает и когда его планируют удалить.
- Критичное новое поведение можно отключить без ручной хирургии по коду.
- Кодовая база не накапливает бессрочные флаги без явной причины.

### AP-059 — Health checks, readiness и деградация зависимостей
**Добавлено:** 2026-03-08 00:20  
**Описание:** Система явно различает "процесс жив", "готов обслуживать трафик" и "работает в деградированном режиме".  
**Почему:**
- Liveness и readiness решают разные задачи и не должны быть слиты в один бессмысленный endpoint.
- Правильная деградация зависимости лучше, чем ложное "все зеленое" или полный self-kill без причины.
- Оркестратор и команда эксплуатации должны получать правдивый сигнал о состоянии сервиса.  
  **Требование:**
- Liveness проверяет жизнеспособность процесса, а readiness — способность безопасно принимать рабочую нагрузку.
- Критичные зависимости учитываются в readiness; некритичные не должны бездумно валить liveness.
- Деградированный режим, если он допустим, должен быть явно определен и наблюдаем.
- Health endpoint не должен скрывать системную неготовность за формальным `200 OK`.  
  **Критерии проверки:**
- Из health semantics понятно, можно ли посылать в инстанс продовый трафик.
- Падение второстепенной зависимости не убивает сервис без причины, если есть допустимая деградация.
- При критичной неготовности оркестратор получает отрицательный readiness-сигнал вовремя.

## База данных и производительность запросов

### AP-060 — Индексы следуют за реальными query patterns
**Добавлено:** 2026-03-08 00:20  
**Описание:** Индексы проектируются под реальные фильтры, сортировки, join paths и hot queries, а не как случайный набор "на всякий случай".  
**Почему:**
- Отсутствующий индекс бьет по latency, а лишний индекс бьет по write cost, вакууму и миграциям.
- Query pattern важнее интуиции о том, "что, наверное, пригодится".
- Индексная дисциплина особенно важна при росте таблиц и concurrency.  
  **Требование:**
- Для hot queries и критичных lookup paths существует обоснованная индексная стратегия.
- Добавление индекса оценивается вместе с его write amplification и operational cost.
- Query shape, под который нужен индекс, должна быть явной и повторяемой, а не гипотетической.
- Полные сканы горячих таблиц в критичном пользовательском пути без явного обоснования считаются дефектом.  
  **Критерии проверки:**
- Для проблемного запроса можно показать, какой индекс его обслуживает или почему он не нужен.
- Новые индексы добавляются под конкретный pattern, а не по интуиции без EXPLAIN-мышления.
- Критичный read/write path не деградирует из-за отсутствия базовой индексной гигиены.

### AP-061 — Безопасные production migrations
**Добавлено:** 2026-03-08 00:20  
**Описание:** Production migrations и backfill-операции выполняются с учетом lock behavior, размера данных и operational impact на работающую систему.  
**Почему:**
- Миграция — это runtime-изменение production-системы, а не просто "ещё один файл в git".
- Большие таблицы, блокировки и backfill-операции ломают прод чаще, чем сама новая бизнес-логика.
- Без дисциплины миграций rollback становится фикцией.  
  **Требование:**
- Production migrations проектируются с учетом размера таблиц, lock behavior, backfill cost и rollback strategy.
- Trivial schema change и operationally heavy migration различаются явно; тяжелые backfill/DDL не маскируются под "обычную миграцию".
- Долгие blocking migrations на горячих таблицах без явного operational plan считаются дефектом.
- Execution plan миграции учитывает окно выполнения, impact на трафик и поведение при rollback/stop.  
  **Критерии проверки:**
- Для нетривиальной миграции можно объяснить порядок rollout и поведение при откате.
- Горячие таблицы не получают неожиданную долгую блокировку без плана.
- Backfill и schema change не маскируются под "обычную маленькую правку".

### AP-062 — Postgres guardrails для OLTP-нагрузки
**Добавлено:** 2026-03-08 00:20  
**Применимость:** если primary OLTP database = Postgres  
**Описание:** Для Postgres-нагрузки задаются базовые защитные привычки, снижающие риск блокировок, runaway queries и деградации под ростом.  
**Почему:**
- Postgres прощает многое на старте и больно наказывает за это позже.
- Большая часть практических проблем приходит не из SQL-синтаксиса, а из долгих транзакций, lock contention и отсутствия guardrails.
- Несколько простых ограничений дают disproportionate operational value.  
  **Требование:**
- Для прикладочной нагрузки задаются разумные `statement_timeout` и `lock_timeout` или их эквивалентная guardrail-политика.
- Долгоживущие транзакции в request path не считаются нормой.
- Горячие запросы анализируются через `EXPLAIN`/план выполнения до того, как проблема станет инцидентом.
- Изменения индексов и тяжелые DDL-операции на больших таблицах выполняются с учетом impact на production workload.  
  **Критерии проверки:**
- Сервис не может бесконечно держать висящий запрос или lock без guardrail.
- Для горячих Postgres-paths есть привычка смотреть на план выполнения, а не гадать.
- Базовые operational risk patterns Postgres учтены до появления аварийного деградационного кейса.
